# GITHUB MCP servers

There are three relevant GitHub MCP options, but practically two live ones. (1) The OFFICIAL github/github-mcp-server is the current, GitHub-maintained server and ships in two flavors that share one codebase: a REMOTE hosted HTTP server at https://api.githubcopilot.com/mcp/ (OAuth or PAT via Authorization: Bearer; GitHub manages runtime/updates; recommended for most users; has remote-only tools like Copilot coding agent), and a LOCAL server you run yourself as a Docker container (ghcr.io/github/github-mcp-server) or native Go binary over stdio, authenticated with a Personal Access Token. Both support fine-grained --toolsets / GITHUB_TOOLSETS scoping, a --read-only / GITHUB_READ_ONLY=1 mode, and dynamic toolset discovery (GITHUB_DYNAMIC_TOOLSETS=1). The remote server additionally exposes per-toolset URLs (e.g. .../mcp/x/issues) and read-only variants by appending /readonly. (2) The OLD reference server @modelcontextprotocol/server-github (npx) / mcp/github (Docker) is DEPRECATED and moved to the archived repo modelcontextprotocol/servers-archived; its README explicitly says development moved to github/github-mcp-server. It still installs and exposes ~26 tools but is unmaintained and should not be chosen for new work. For a pi MCP bridge: stdio (Docker/binary) and http (remote) are the two transports to bridge; pick based on whether you want a hosted endpoint (http) or a self-hosted container (stdio). NOTE: tool names below are quoted from the live README; older single-purpose names (e.g. get_issue, create_issue, get_pull_request, get_pull_request_diff) have been consolidated into issue_read/issue_write/pull_request_read with old names kept as backward-compat aliases.

## Recommendation
Adopt the OFFICIAL github/github-mcp-server. Start with the REMOTE hosted HTTP server (https://api.githubcopilot.com/mcp/) bridged as an http MCP server — it needs zero local runtime, GitHub handles updates, supports OAuth or a Bearer PAT, and gives per-toolset URLs plus /readonly scoping out of the box; it is the lowest-ops way to give a pi agent GitHub access. If pi's MCP bridge cannot do http/OAuth, or you need to run inside a locked-down/offline environment, fall back to the LOCAL Docker server (ghcr.io/github/github-mcp-server) over stdio with a PAT and GITHUB_TOOLSETS/GITHUB_READ_ONLY scoping. In both cases, start read-only (append /readonly or set GITHUB_READ_ONLY=1) and enable only the toolsets you need (e.g. repos,issues,pull_requests) to limit tool count and blast radius. Avoid the deprecated @modelcontextprotocol/server-github entirely — it is archived and its own README points to the official server.

## Servers

### GitHub MCP Server — Remote (hosted HTTP)  [official/verified-quoted]
- source: Hosted by GitHub at https://api.githubcopilot.com/mcp/ (built from the github/github-mcp-server repo as a library bound into GitHub infrastructure)
- transport: http | auth: OAuth (point client at URL and authorize once) OR Personal Access Token sent as 'Authorization: Bearer <PAT>' header. PAT scopes: docs recommend minimum necessary, e.g. repo, read:packages, read:org for team access.
- key tools: get_file_contents, search_code, list_issues, add_issue_comment, issue_read, issue_write, list_pull_requests, create_pull_request, pull_request_read, search_pull_requests, merge_pull_request, create_pull_request_with_copilot (remote-only)
- piConfig: Bridge as an HTTP MCP server: declare url=https://api.githubcopilot.com/mcp/ with an Authorization: Bearer <PAT> header (or OAuth if the bridge supports it). In an mcp.json / pi-mcp-adapter entry: {"github":{"transport":"http","url":"https://api.githubcopilot.com/mcp/","headers":{"Authorization":"Bearer ${GITHUB_MCP_PAT}"}}}. To scope, point at a per-toolset URL (e.g. .../mcp/x/issues/readonly) or send X-MCP-Toolsets / X-MCP-Readonly headers. NOTE: exact pi adapter key names are not grounded in this repo (template scaffold has no MCP config); shape follows standard MCP http-server config.
- src: https://github.com/github/github-mcp-server/blob/main/docs/remote-server.md

### GitHub MCP Server — Local (Docker / native binary, stdio)  [official/verified-quoted]
- source: Docker image ghcr.io/github/github-mcp-server (also distributed as a native Go binary github-mcp-server)
- transport: stdio | auth: Personal Access Token via GITHUB_PERSONAL_ACCESS_TOKEN env var. Scopes: minimum necessary, e.g. repo, read:packages, read:org for team access.
- key tools: get_file_contents, search_code, list_issues, add_issue_comment, issue_read, issue_write, list_pull_requests, create_pull_request, pull_request_read, merge_pull_request, create_or_update_file, push_files, create_branch, search_repositories, get_commit
- piConfig: Bridge as a stdio MCP server. mcp.json / pi-mcp-adapter entry: {"github":{"transport":"stdio","command":"docker","args":["run","-i","--rm","-e","GITHUB_PERSONAL_ACCESS_TOKEN","ghcr.io/github/github-mcp-server"],"env":{"GITHUB_PERSONAL_ACCESS_TOKEN":"${GITHUB_PAT}"}}}. For the native binary use command=/path/to/github-mcp-server, args=["stdio"]. Add GITHUB_TOOLSETS / GITHUB_READ_ONLY=1 / GITHUB_DYNAMIC_TOOLSETS=1 to env to scope. NOTE: pi adapter key names not grounded in this repo; shape follows standard MCP stdio-server config.
- src: https://github.com/github/github-mcp-server

### Reference GitHub server — @modelcontextprotocol/server-github (DEPRECATED)  [experimental/verified-quoted]
- source: npm: @modelcontextprotocol/server-github (npx) and Docker image mcp/github. Now in modelcontextprotocol/servers-archived.
- transport: stdio | auth: Personal Access Token via GITHUB_PERSONAL_ACCESS_TOKEN. Scopes: repo (or public_repo for public-only access).
- key tools: create_or_update_file, push_files, search_repositories, create_repository, get_file_contents, create_issue, create_pull_request, fork_repository, list_issues, search_code, merge_pull_request
- piConfig: Bridge as stdio MCP server: {"github":{"transport":"stdio","command":"npx","args":["-y","@modelcontextprotocol/server-github"],"env":{"GITHUB_PERSONAL_ACCESS_TOKEN":"${GITHUB_PAT}"}}}. DO NOT adopt for new work — it is archived/unmaintained; use the official server above. No toolset-scoping or read-only flags.
- src: https://github.com/modelcontextprotocol/servers-archived/tree/main/src/github

# BROWSER MCP servers

Five real, current browser-automation MCP servers were evaluated for connecting to the pi coding agent. The two strongest options are official and actively maintained: **Playwright MCP** (microsoft/playwright-mcp, npm `@playwright/mcp`) and **Chrome DevTools MCP** (ChromeDevTools/chrome-devtools-mcp, npm `chrome-devtools-mcp`). Both default to driving a fresh browser, can attach to an existing Chrome/profile (Playwright via `--cdp-endpoint`/`--extension`/`--user-data-dir`; DevTools via `--browser-url`), and both lead with an accessibility-tree text snapshot (`browser_snapshot` / `take_snapshot`) for cheap, deterministic interaction rather than pixel screenshots. Playwright MCP is cross-browser (Chromium/Firefox/WebKit/Edge) and tuned for navigation+testing ('Uses Playwright's accessibility tree, not pixel-based input'; 'No vision models needed'). Chrome DevTools MCP is Chrome-only but adds deep debugging — network inspection, console with source-mapped stacks, performance traces, Lighthouse, heap snapshots ('Recording traces and extracting actionable performance insights'); built on Puppeteer, needs Node v20.19+. The classic **Puppeteer reference server** (`@modelcontextprotocol/server-puppeteer`) is ARCHIVED/deprecated (moved to modelcontextprotocol/servers-archived) and ecosystem guidance points to Playwright MCP; it is screenshot-only (no a11y snapshot) and launches a fresh browser ('the NPX version will open a browser window'). Avoid for new work. **Browser MCP** (`@browsermcp/mcp` v0.1.3, browsermcp.io) is a community server+Chrome-extension combo whose whole point is driving YOUR already-open, logged-in Chrome profile over a local WebSocket — useful for authenticated/stealth research but early (the repo 'currently cannot yet be built on its own'). **browser-use** offers an open-source stdio MCP server (`uvx browser-use --mcp`) plus a hosted cloud MCP; it is a higher-level agent (it can run its own autonomous sub-agent via `retry_with_browser_use_agent`) and needs its own LLM API key, so it overlaps with pi rather than being a pure tool layer. All transports/commands below are quoted from official READMEs/tool-reference/package.json; the pi repo has no existing MCP config convention (it's a template), so piConfig uses the standard mcpServers JSON block these servers document. Confidence is 'verified-quoted' where commands/tool lists came straight from raw GitHub sources; npm 403'd, so Browser MCP's run config is marked 'likely'.

## Recommendation
Adopt **Playwright MCP** (`npx @playwright/mcp@latest`) first as the default driver, and add **Chrome DevTools MCP** (`npx -y chrome-devtools-mcp@latest`) as the second. Rationale: both are official and actively maintained, run as zero-auth stdio servers that drop straight into pi's mcpServers config, and both expose an accessibility-tree text snapshot (browser_snapshot / take_snapshot) so the agent interacts via cheap deterministic element IDs instead of burning tokens on screenshots. Playwright MCP covers the broad navigate/click/type/evaluate/network/screenshot surface cross-browser and is the explicit successor to the deprecated Puppeteer server -- best general-purpose pick for research and UI testing. Chrome DevTools MCP is the complement when the agent needs real debugging: network-request inspection, console with source-mapped stacks, performance traces, and Lighthouse -- exactly the 'inspect network/console' part of the ask. Both can also attach to an existing logged-in Chrome (Playwright --cdp-endpoint/--extension, DevTools --browser-url) when authenticated sites are needed. Skip the Puppeteer reference server (archived/deprecated). Consider Browser MCP only if driving the user's real logged-in Chrome profile is a hard requirement, and browser-use only if you want a built-in autonomous browsing agent (it needs its own LLM key and overlaps pi's own agent loop).

## Servers

### Playwright MCP  [official/verified-quoted]
- source: npm: @playwright/mcp (repo: microsoft/playwright-mcp); also Docker mcp/playwright
- transport: stdio | auth: none for the server; browser sessions authenticated via persistent profile (--user-data-dir) or by attaching to an existing browser (--extension / --cdp-endpoint)
- key tools: browser_navigate, browser_navigate_back, browser_click, browser_type, browser_fill_form, browser_select_option, browser_hover, browser_drag, browser_press_key, browser_snapshot, browser_take_screenshot, browser_evaluate, browser_console_messages, browser_network_requests, browser_handle_dialog, browser_file_upload, browser_wait_for, browser_tabs, browser_resize, browser_close
- piConfig: { "mcpServers": { "playwright": { "command": "npx", "args": ["@playwright/mcp@latest"] } } }  -- append e.g. "--headless", "--browser","chromium", or "--cdp-endpoint","http://localhost:9222" to args; equivalently a stdio pi-mcp-adapter entry pointing at the same command. Accessibility-snapshot-first (browser_snapshot returns structured a11y tree, by default only changed parts on subsequent calls; browser_take_screenshot is the pixel fallback). Drives a FRESH browser by default ('--headless run browser in headless mode, headed by default'); ATTACH via '--extension Connect to a running browser instance (Edge/Chrome only)', '--cdp-endpoint <endpoint>', '--user-data-dir <path>', or '--isolated keep the browser profile in memory, do not save it to disk'. Cross-browser chromium/firefox/webkit/msedge. browser_evaluate runs page JS; optional tool groups via --caps (storage/network/devtools/vision/pdf/testing). 40+ tools total.
- src: https://github.com/microsoft/playwright-mcp

### Chrome DevTools MCP  [official/verified-quoted]
- source: npm: chrome-devtools-mcp (repo: ChromeDevTools/chrome-devtools-mcp); built on Puppeteer
- transport: stdio | auth: none for the server; attach to a logged-in Chrome via --browser-url to reuse existing auth/sessions
- key tools: navigate_page, new_page, select_page, list_pages, close_page, click, type_text, fill, fill_form, hover, drag, press_key, upload_file, click_at, take_snapshot, take_screenshot, evaluate_script, list_console_messages, get_console_message, list_network_requests, get_network_request, performance_start_trace, performance_stop_trace, performance_analyze_insight, lighthouse_audit, emulate, resize_page, wait_for
- piConfig: { "mcpServers": { "chrome-devtools": { "command": "npx", "args": ["-y", "chrome-devtools-mcp@latest"] } } }  -- append e.g. "--headless", "--isolated", or "--browser-url=http://127.0.0.1:9222" to args; equivalently a stdio pi-mcp-adapter entry. Chrome-only, debugging-heavy (network inspection, console with source-mapped stacks, performance traces, Lighthouse). take_snapshot is the a11y text snapshot ('Take a text snapshot of the currently selected page based on the a11y tree', elements get unique identifiers) vs take_screenshot (PNG/JPEG/WebP). Drives a fresh Chrome by default; attach to an EXISTING Chrome via '--browser-url=http://127.0.0.1:9222' (or --wsEndpoint); pick channel via '--channel' (canary/dev/beta/stable); '--isolated' temporary profile auto-cleaned. Extra tool groups: heap snapshots (take_heapsnapshot etc.), extensions, WebMCP. Requires Node.js v20.19+. 26 core tools.
- src: https://github.com/ChromeDevTools/chrome-devtools-mcp

Listed only for legacy parity/completeness.

### Browser MCP  [community/likely]
- source: npm: @browsermcp/mcp (repo: BrowserMCP/mcp; site browsermcp.io) + companion Chrome extension
- transport: stdio (bridges to the Chrome extension over a local WebSocket) | auth: none in the server; by design reuses your existing browser profile so you stay 'logged into all your services'
- key tools: (tool names not enumerated in README; documented capabilities: navigate, click, type, screenshot, read page, automate the attached profile)

### browser-use (MCP server)  [community/verified-quoted]
- source: PyPI/CLI: browser-use[cli] (repo: browser-use/browser-use); also a hosted Cloud MCP
- transport: stdio (local open-source); http (hosted Cloud MCP) | auth: local server needs your own LLM API key (e.g. OPENAI_API_KEY env var); Cloud MCP uses a browser-use account/API key
- key tools: browser_navigate, browser_click, browser_type, browser_get_state, browser_extract_content, browser_get_html, browser_screenshot, browser_scroll, browser_go_back, browser_list_tabs, browser_switch_tab, browser_close_tab, retry_with_browser_use_agent, browser_list_sessions, browser_close_session, browser_close_all

Higher-level than the others: built on Playwright/Chromium and ships an autonomous agent -- note retry_with_browser_use_agent runs browser-use's OWN LLM-driven sub-agent, so it overlaps with pi rather than being a pure low-level tool layer. Mixes screenshot (browser_screenshot) and DOM extraction (browser_get_state / browser_extract_content / browser_get_html). Requires bringing your own LLM key. Tool list verified from browser_use/mcp/server.py; run command verified from official docs.
- src: https://docs.browser-use.com/open-source/customize/integrations/mcp-server
